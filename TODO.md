# To-do

- [ ] **Replace the engine `sys.path` bridge with a real package dependency.** The backend reaches the engine's leaf modules (`chain`/`load_index`/`retriever`) by computing the sibling repo's path and prepending it to `sys.path` (`ENGINE_DIR = UMBRELLA / "engine-rag-context-pipeline"` + `sys.path.insert(...)` in `backend-rag-context-pipeline/api/main.py`, hence the `# noqa: E402` on the late imports) rather than importing an installed package. This couples the two repos to an exact side-by-side on-disk layout, pins no version, isn't `pip`-installable in isolation, and forces the Docker build to use the umbrella root as its context (+ a CI step that checks out the sibling engine repo). Migration: (1) package the engine with a `pyproject.toml` under a proper namespace (e.g. `from rag_engine.chain import build_chain`) — note this conflicts with the engine's deliberately flat, `__init__.py`-free layout, so it's a real restructure; (2) publish it (PyPI / private index / Git URL / editable `pip install -e`); (3) add it to the backend's `requirements.txt`; (4) delete the `sys.path` bridge + `ENGINE_DIR` and move the three imports to the top of the file as ordinary imports. Afterward the Dockerfile no longer needs the umbrella-root build context and the CI sibling-checkout step goes away.

- [x] **Streaming — done via `POST /chat`.** The backend now streams answers as raw `text/plain` fragments (`StreamingResponse` + `chain.stream` in `backend-rag-context-pipeline/api/main.py`), built to the contract in `integration-plan.md`. `/ask` deliberately stays blocking JSON for its existing consumers.

- [ ] **Multi-turn chat support — `POST /chat` currently ignores prior turns.** The endpoint accepts the full conversation (`ChatRequest.messages`) but only answers the last user message: `query = req.messages[-1].content` in `backend-rag-context-pipeline/api/main.py`, fed to a single-question chain, so follow-ups with pronouns/ellipsis ("what are the penalties for *it*?") lose their referent and retrieval embeds an under-specified query. Two ways to add memory: (a) **query rewriting / condensation** — a cheap LLM pass that folds the recent history + latest message into one self-contained question *before* retrieval (keeps the retriever and answer chain single-question; best for retrieval quality); or (b) **history in the prompt** — thread prior turns into the answer chain's prompt so generation is context-aware (doesn't fix retrieval on its own). (a) and (b) compose. Scope: keep the `messages` contract unchanged (frontend already sends full history), bound the history window, and decide whether to retrieve per-turn or reuse prior context. Update the `/chat` docstring's "v1 ignores prior turns" note and add a backend test covering a follow-up turn.

- [ ] **`/ask` holds a pooled DB connection across the LLM call.** In `backend-rag-context-pipeline/api/main.py` the `ask` handler runs `chain.invoke(...)` *inside* the `with app.state.pool.connection()` scope, so a pooled Postgres connection is held for the entire multi-second OpenAI generation even though the DB is only needed for the fast retrieval step. Under concurrency this can exhaust the pool with connections idling on OpenAI. Fix: mirror `/chat` — retrieve inside the `with`, then run the LLM call outside it (no contract change; `/ask` stays blocking JSON). Note: this is the server-side concurrency fix, *not* streaming `/ask` — streaming is intentionally `/chat`'s job. Optionally consider `async def` + `run_in_threadpool` for the blocking LLM call, but that's likely over-engineering at this corpus's scale.

- [ ] **Answer-quality scoring is a deliberate next increment.** The eval (`engine-rag-context-pipeline/eval/run_eval.py`) currently scores retrieval only (hit-rate@k / MRR). Build on the ground-truth `answer` field already in `eval/dataset.jsonl` to score generated answers against the reference.

- [ ] **Put the HTTP API behind the optional API-key gate.** To enable the shared-secret gate on the public endpoints, see the backend repo's README (`backend-rag-context-pipeline/README.md`, the `RAG_API_KEY` env var): when set, `/ask` and `/chat` require the `X-API-Key: <value>` header (401 otherwise); `/health` stays open and auto docs are disabled.

- [ ] **Make container startup fully offline (`HF_HUB_OFFLINE=1`).** The eval run inside the engine container warned `You are sending unauthenticated requests to the HF Hub` — sentence-transformers still pings HuggingFace at startup to check for model updates, even though the BGE weights are baked into the image. That adds a network round-trip and, worse, a network dependency: on a flaky connection or if HF is down, startup stalls or fails. Fix: set `HF_HUB_OFFLINE=1` in the compose `environment:` block of `engine-rag-context-pipeline/docker-compose.yml` (and `backend-rag-context-pipeline/docker-compose.yml`, which has the same baked-model setup).
